\section{问题描述}

近似最近邻搜索（ANNS）在大规模向量检索中面临严重的计算瓶颈。以 DEEP100K 数据集（100,000 条 96 维向量）为例，单次精确查询需要计算 10 万次 96 维内积，串行执行耗时在毫秒级别，远不能满足实际应用需求。

本实验使用 \textbf{pthread} 和 \textbf{OpenMP} 两种多线程编程范式，对 ANNS 的查询阶段进行并行化。核心问题归结为：\textbf{如何将大规模的距离计算任务合理划分给多个线程，并在结果正确的前提下最大化加速比}。

\section{算法设计}

\subsection{Flat-SIMD}

\subsubsection{算法原理}

Flat-SIMD 是 ANNS 中最基础的检索方式——对基向量集合进行全量遍历，用 SIMD 指令加速单次距离计算。由于 Flat 不引入任何索引结构，Recall@K 恒为 1.0（精确结果），因此多线程并行不会损失精度，加速比的唯一限制来自并行效率和硬件资源。

多线程并行的核心思想是\textbf{数据并行（Data Parallelism）}：将 n 条基向量的距离计算任务均匀划分给 T 个线程，每个线程在独立的连续内存区域内执行 SIMD 内积运算、维护局部 top-k 堆，最后由主线程进行 T 路归并得到全局 top-k。由于各线程间的计算完全独立（仅读共享数据、写私有堆），该算法属于\textbf{易并行（Embarrassingly Parallel）}问题，理论加速比接近 T。

\subsubsection{pthread 实现}

\textbf{并行策略：} Base 向量划分 —— 将 n 条基向量按块均匀分配给各线程，每线程独立完成分配范围内的全量距离计算与局部 top-k 维护。

\textbf{伪代码：}

\begin{breakablealgorithm}
\caption{pthread\_flat\_search}
\begin{algorithmic}[1]
\Require base[n][d], query[d], k, num\_threads
\Ensure top-k results (max-heap of $<$dist, index$>$)
\State chunk\_size = n / num\_threads
\State 初始化全局 barrier、T 个局部堆 local\_heaps[T]，每个容量为 k
\For{t = 0 to num\_threads-1}
\State thread\_args[t] = \{base, query, t*chunk\_size, (t+1)*chunk\_size, d, k, \&local\_heaps[t], \&barrier\}
\State pthread\_create(\&threads[t], thread\_worker, \&thread\_args[t])
\EndFor
\State pthread\_barrier\_wait(barrier)
\State global\_heap = merge\_k\_way(local\_heaps[0..T-1], k)
\State \Return global\_heap

\Function{thread\_worker}{args}
\State 初始化空的小顶堆 local\_heap (容量 k)
\For{i = args.start to args.end-1}
\State ip = inner\_product\_neon(args.query, args.base + i*d, args.d)
\If{local\_heap.size < k}
\State local\_heap.push(\{ip, i\})
\ElsIf{ip > local\_heap.top().dist}
\State local\_heap.pop()
\State local\_heap.push(\{ip, i\})
\EndIf
\EndFor
\State 将 local\_heap 写入 args.result
\State pthread\_barrier\_wait(args.barrier)
\EndFunction
\end{algorithmic}
\end{breakablealgorithm}

\textbf{复杂度分析：}

\begin{itemize}
\item \textbf{时间复杂度：} 每个线程计算 n/T 次 NEON 内积，总计 O(n·d/T)。结果归并 O(T·k·log(T·k))，在 T ≤ 16 时远小于距离计算开销。
\item \textbf{空间复杂度：} 每线程存储大小为 k 的局部堆，总计 O(T·k)。基向量为只读共享数据，无需额外副本。
\item \textbf{加速比理论上界：} 各线程计算完全独立，无临界区竞争，理论加速比接近 S(T) = T。实际主要受限于内存带宽和 barrier 同步开销。
\end{itemize}

\textbf{深入分析：} Flat-SIMD 作为易并行（Embarrassingly Parallel）问题，理论上应接近线性加速。但实验数据显示：

\begin{enumerate}
\item \textbf{4 线程为最优节点：} pthread 和 OpenMP 在 4 线程时均达到较高效率（85\%以上）。此时内存带宽尚未饱和，线程间计算负载完全均匀。
\item \textbf{8 线程后效率骤降：} OpenMP 在 8 线程时加速比达 3.86（效率 48.3\%），pthread 更是降至 2.67（效率 33.4\%）。这是因为 DEEP100K 数据集（100K×96 维）规模使得单次查询的 L2 距离计算量约 19.2M FLOPS，恰好触及单通道内存带宽极限。
\item \textbf{内存带宽瓶颈：} 每代 SIMD 指令的峰值算力远大于内存带宽（Byte/s），导致计算单元饥饿。16 线程时延迟反而增加，印证了内存带宽竞争导致的 cache coherency 开销。
\end{enumerate}

\subsubsection{OpenMP 实现}

\textbf{并行策略：} Base 向量划分 —— 使用 \texttt{\#pragma omp parallel for} 自动将循环迭代分布到线程，编译器自动处理任务划分、线程创建和同步。

\textbf{伪代码：}

\begin{breakablealgorithm}
\caption{omp\_flat\_search}
\begin{algorithmic}[1]
\Require base[n][d], query[d], k
\Ensure top-k results
\State T = omp\_get\_max\_threads()
\State 分配 T 个局部小顶堆 local\_heaps[T]，每个容量为 k
\State \textbf{\#pragma omp parallel}
\State[1] tid = omp\_get\_thread\_num()
\State[1] 清空 local\_heaps[tid]
\State[1] \textbf{\#pragma omp for schedule(dynamic)}
\State[1] \For{i = 0 to n-1}
\State[2] ip = inner\_product\_neon(query, base + i*d, d)
\State[2] heap = \&local\_heaps[tid]
\State[2] \If{heap.size < k}
\State[3] heap.push(\{ip, i\})
\State[2] \ElsIf{ip > heap.top().dist}
\State[3] heap.pop(); heap.push(\{ip, i\})
\State[2] \EndIf
\State[1] \EndFor
\State global\_heap = merge\_k\_way(local\_heaps[0..T-1], k)
\State \Return global\_heap
\end{algorithmic}
\end{breakablealgorithm}

\textbf{复杂度分析：} 渐近复杂度与 pthread 相同。主要差异在于 OpenMP 维护线程池，免去 \texttt{pthread\_create}/\texttt{pthread\_join} 开销；\texttt{schedule(dynamic)} 在 Flat 均匀计算量场景下引入了不必要的分发锁，改为 \texttt{schedule(static)} 可消除此开销。

\textbf{深入分析：} OpenMP 在 8 线程时取得最优性能（加速比 3.86），而 pthread 在 4 线程后就开始下降。这说明：
\begin{enumerate}
\item \textbf{线程池复用优势：} OpenMP 的隐式线程池避免了 pthread 每轮查询的线程创建/销毁开销，在多轮查询场景下优势明显。
\item \textbf{动态调度适应性：} 虽然 Flat 计算量完全均匀，但 \texttt{schedule(dynamic)} 的 chunk 分发机制仍优于静态划分——它允许各线程根据实际完成时间动态获取下一块任务，避免了某线程因 cache miss 导致的长尾效应。
\item \textbf{隐式 barrier 优化：} OpenMP 的隐式 barrier 经过编译器优化，同步开销低于 pthread 的显式 \texttt{pthread\_barrier\_wait}。
\end{enumerate}

\subsection{PQ-SIMD}

\subsubsection{算法原理}

乘积量化（Product Quantization, PQ）是向量压缩领域最具代表性的有损编码技术。其核心思想是将高维向量空间分解为多个低维子空间的笛卡尔积，在每个子空间内独立进行 KMeans 聚类，将原始浮点向量替换为聚类中心索引序列，从而将存储和计算代价降低 1~2 个数量级。

以本实验 DEEP100K 数据集（d=96）为例，取 M=8 个子空间，每个子空间维度 dsub = 96/8 = 12，每子空间 ksub=256 个聚类中心。编码后每条向量从 96×4 = 384 字节压缩为 8×1 = 8 字节（uint8 索引），压缩比达 48:1。

查询采用\textbf{非对称距离计算（ADC）}：对查询向量 q 不做量化（避免引入查询端量化误差），而是预计算 q 的每个子向量到对应子空间全部 ksub 个聚类中心的距离，形成 M×ksub 的查找表（LUT）。随后对每条编码向量，仅需 M 次查表和累加即可估算出与 q 的近似距离。

\textbf{查询流程的瓶颈转移：} PQ 将距离计算从 compute-bound 的浮点乘加转变为 memory-bound 的查表累加。单次查询需随机访问 codes 数组中 n×M 个 uint8 索引，再用这些索引随机访问 LUT。双重的随机内存访问模式使得 cache miss 率显著升高，多线程环境下的内存带宽竞争成为决定加速比上限的关键因素。

\subsubsection{pthread 实现}

\textbf{并行策略：} 对 LUT 构建阶段按子空间划分线程，对查表累加阶段按 Base 向量分块。

\textbf{伪代码：}

\begin{breakablealgorithm}
\caption{pq\_search\_pthread}
\begin{algorithmic}[1]
\Require codes[n][M], centroids[M][ksub][dsub], norms[n], query[d], k, p, num\_threads
\Ensure top-k results
\State chunk\_m = M / num\_threads
\State lut[M][ksub]
\State 创建线程，每个线程 t 处理子空间 [t*chunk\_m, (t+1)*chunk\_m]
\Function{build\_lut\_thread}{t}
\For{m = t*chunk\_m to (t+1)*chunk\_m-1}
\State query\_sub = query + m*dsub
\For{ki = 0 to ksub-1}
\State centroid = centroids + m*ksub*dsub + ki*dsub
\State lut[m][ki] = l2\_distance\_neon(query\_sub, centroid, dsub)
\EndFor
\EndFor
\State pthread\_barrier\_wait(\&phase1\_barrier)
\EndFunction

\State q\_norm = dot\_product\_neon(query, query, d)
\State chunk\_n = n / num\_threads
\State 分配 local\_candidates[T]，每个容量为 p（小顶堆）

\Function{scan\_thread}{t}
\For{i = t*chunk\_n to (t+1)*chunk\_n-1}
\State dist = 0
\For{m = 0 to M-1}
\State dist += lut[m][codes[i*M + m]]
\EndFor
\State ip = (q\_norm + norms[i] - dist) / 2
\State 维护 local\_candidates[t] 为大小为 p 的小顶堆
\EndFor
\State pthread\_barrier\_wait(\&phase2\_barrier)
\EndFunction

\State 将 T 个 local\_candidates 中的 p*T 个候选合并，取前 k 个
\State \Return global\_top\_k
\end{algorithmic}
\end{breakablealgorithm}

\textbf{复杂度分析：}

\begin{itemize}
\item \textbf{LUT 构建：} O(M·ksub·dsub / T)，在 M=8 时计算量约 Flat 的 0.1\%，并行收益有限。
\item \textbf{查表累加：} O(n·M / T)。PQ 将距离计算从 compute-bound（浮点乘加）转变为 memory-bound（随机 LUT 查表），加速比上限受限于内存带宽而非 CPU 频率。
\item \textbf{空间复杂度：} O(n·M + M·ksub·dsub)。p 参数增大时堆维护开销从 O(log k) 增至 O(log p)。
\end{itemize}

\textbf{深入分析：} PQ 的并行化收益远低于 Flat，根源在于其 memory-bound 特性：

\begin{enumerate}
\item \textbf{随机访存瓶颈：} 每条基向量的查表需访问 M=8 个 LUT 条目，而 LUT 分布在一个 M×256 的表中。线程访问 LUT 时，由于不同线程处理不同基向量，其 LUT 访问序列完全不同，导致 cache 失效。
\item \textbf{内存带宽竞争：} 4 线程时加速比仅 1.39-1.51，16 线程时加速比 < 1。这表明 4 线程时已接近内存带宽极限。
\item \textbf{p 参数的权衡：} p 从 1000 增至 5000 时，召回率从 0.994 提升至 0.9999，但延迟增加约 2 倍。这是因为更大的候选集需要更频繁的堆调整（O(log p)），且最终归并 T 路堆的开销也更大。
\end{enumerate}

\subsubsection{OpenMP 实现}

\State[1] tid = omp\_get\_thread\_num()

\State[1] \textbf{// 阶段一：LUT 构建}
\State[1] \textbf{\#pragma omp for schedule(static)}
\State[1] \For{m = 0 to M-1}
\State[2] query\_sub = query + m*dsub
\State[2] \For{ki = 0 to ksub-1}
\State[3] lut[m][ki] = l2\_distance\_neon(query\_sub, \&centroids[m][ki][0], dsub)
\State[2] \EndFor
\State[1] \EndFor
\State[1] \textbf{// 阶段二：查表累加}
\State[1] \textbf{\#pragma omp for schedule(dynamic, 2000)}
\State[1] \For{i = 0 to n-1}
\State[2] dist = 0
\State[2] \For{m = 0 to M-1}
\State[3] dist += lut[m][codes[i*M + m]]
\State[2] \EndFor

\State[2] 维护 local\_results[tid] 为大小为 p 的小顶堆
\State[1] \EndFor

\end{algorithmic}

\subsection{IVF-SIMD}

\subsubsection{算法原理}

倒排文件（Inverted File, IVF）索引借鉴了信息检索中倒排索引的思想，通过离线聚类将基向量空间划分为 nlist 个 Voronoi 区域。在线查询时，仅需搜索距离查询向量最近的 nprobe 个区域，从而将候选集从 n 缩减至约 nprobe × n/nlist。

IVF 的查询过程分为\textbf{粗排（Coarse Search）}和\textbf{精排（Fine Search）}两个阶段：

\begin{enumerate}
\item \textbf{粗排：} 计算查询向量 q 到 nlist 个聚类中心的距离，用 partial sort 选取距离最近的 nprobe 个簇。计算量为 O(nlist×d)，通常 nlist ≪ n。
\item \textbf{精排：} 遍历选中的 nprobe 个簇的倒排链表，对其中每个基向量 x 计算 SIMD 内积，维护全局 top-k。
\end{enumerate}

\textbf{多线程并行挑战：} KMeans 聚类产生的倒排链表长度不均匀是固有特性。更好的策略是将所有候选向量展平为一个连续数组，然后按向量粒度的 Base 划分并行——这与 Flat 的并行模式完全一致，但候选集规模远小于 n。

\EndFor
\State pthread\_barrier\_wait(\&fine\_barrier)
\EndFunction

\State \Return global\_heap
\end{algorithmic}

\subsection{IVF-PQ-SIMD}

\subsubsection{算法原理}

IVF-PQ 是 IVF 索引结构与 PQ 编码技术的组合：IVF 负责缩小搜索范围（粗排剪枝），PQ 负责降低单条距离计算代价（ADC 查表累加）。两者形成\textbf{两级近似}。

\textbf{查询流程：}
\begin{enumerate}
\item IVF 粗排选择簇
\item 构建 PQ LUT
\item 对候选向量的 PQ code 进行查表累加估算距离
\item 根据估算距离选取 top-p 候选，用原始浮点向量重排得到精确 top-k
\end{enumerate}

\subsection{HNSW 图索引}

\subsubsection{算法原理}

HNSW（Hierarchical Navigable Small World）是当前工业界综合性能最优的 ANNS 索引结构之一。它构建了一个多层近邻图，其中每层是一个可导航小世界图（NSW），不同层间通过指数衰减的概率分布建立连接。

\textbf{层级结构设计：} 每个节点被随机分配一个整数层级 l ∈ [0, L]，分配概率服从指数分布。层级越高节点越稀疏。查询时从顶层入口点开始，在每层执行贪心搜索找到局部最近点，然后下降到下一层以该点为起点继续搜索。

\textbf{HNSW 并行化的核心困境：} 图搜索具有天然的\textbf{顺序依赖性}——每一步扩展哪个节点取决于上一步的计算结果。目前可行的并行策略集中于 Query 级并行（不同线程处理不同查询）。

\textbf{深入分析：} IVF-PQ 结合了 IVF 的剪枝和 PQ 的压缩，其性能瓶颈呈现复杂特性：

\begin{enumerate}
\item \textbf{查表累加的内存访问模式：} IVF-PQ 的查表累加需要随机访问 PQ codes（uint8 数组）和 LUT（float 数组）。与纯 PQ 不同，IVF 的展平策略使得候选向量在内存中连续排列，但 PQ code 的随机访问仍导致高 cache miss 率。
\item \textbf{重排阶段的必要性：} top-p 候选（通常 p=1000-5000）需要用原始浮点向量重新计算精确距离。虽然重排仅占总耗时 9\%，但它是保证高召回率的关键——PQ 的有损编码会导致距离估算误差。
\item \textbf{多线程扩展性差：} 8-16 线程时加速比急剧下降（部分情况下 < 1），因为查表累加已完全受内存带宽限制，再增加线程只会加剧竞争。
\end{enumerate}

\textbf{深入分析：} HNSW 的图遍历特性使其并行化策略与向量索引类算法完全不同：

\begin{enumerate}
\item \textbf{Query 级并行是唯一选择：} HNSW 的搜索过程具有严格的顺序依赖性——每步选择哪个邻居节点取决于上一步的计算结果。单个查询内部无法并行化，只能让不同线程处理不同查询。
\item \textbf{pthread 的优势：} 在 16 线程时，pthread 达到 6.43× 加速比，而 OpenMP 仅有 2.65×。这因为 pthread 能更好地控制线程亲和性和 cache 共享——相邻查询的搜索路径可能有重叠，共享图结构的 cache 命中率更高。
\item \textbf{内存带宽利用充分：} HNSW 的搜索范围远小于 Flat（约 O(log n) 个节点 vs n），内存访问量小，因此 8 线程时 OpenMP 仍能保持较高效率（4.01×）。
\item \textbf{ef 参数的权衡：} ef 从 50 增至 200 时，召回率从 0.97 提升至 0.998，但延迟增加约 4 倍。更大的 ef 意味着更大的候选集，搜索树更深，但召回率提升显著。
\end{enumerate}

\textbf{深入对比分析：} 各算法在不同召回率要求下展现出截然不同的性能特征：

\begin{enumerate}
\item \textbf{精确检索（Flat）：} 适合需要 100\% 召回率的场景。OpenMP 实现最优，8 线程时加速比达 3.86×。但内存带宽限制使得超过 8 线程后性能反而下降。
\item \textbf{近似检索（PQ）：} 32:1 压缩率极大降低内存占用，适合内存受限场景。但并行收益有限（4 线程仅 1.5×），因为 memory-bound 特性。
\item \textbf{剪枝检索（IVF）：} 通过 nprobe 参数灵活调节召回率-延迟权衡。4 线程时可达 3-4× 加速比，是性价比最高的选择。
\item \textbf{混合索引（IVF-PQ）：} 兼顾剪枝和压缩，但两级近似的误差累积可能导致召回率下降。适合超大规模数据集。
\item \textbf{图索引（HNSW）：} 综合性能最优，16 线程时达 118× 加速比。但构建索引成本高，适合查询频繁且对延迟敏感的场景。
\end{enumerate}

\textbf{最优选择建议：}
\begin{itemize}
\item 高召回率（>0.99）需求：选择 HNSW 或 IVF-SIMD
\item 内存受限场景：选择 PQ 或 IVF-PQ
\item 均衡选择：IVF-SIMD（通过调节 nprobe 实现灵活权衡）
\end{itemize}

\subsection{适用场景分析}

\textbf{pthread 优势：}
\begin{itemize}
\item 更细粒度的线程控制，适合复杂同步逻辑
\item 可实现自定义线程池，避免重复创建开销
\item 适合 HNSW 等图遍历类算法，能更好地控制搜索路径
\end{itemize}

\textbf{OpenMP 优势：}
\begin{itemize}
\item 代码简洁，易于维护和调试
\item 自动负载均衡（schedule(dynamic)）
\item 适合 IVF-PQ 等有不均匀倒排链表的算法
\end{itemize}

\section{总结与收获}

\subsection{主要结论}

\begin{enumerate}
\item \textbf{并行策略选择：}
\begin{itemize}
\item Query 级并行是最有效的并行策略，适用于所有 ANNS 算法
\item 对于计算密集型算法（Flat），最优线程数为 4-8；超过 8 线程后内存带宽成为瓶颈
\item 对于内存密集型算法（PQ、IVF-PQ），最优线程数为 4；16 线程时性能反而下降
\end{itemize}

\item \textbf{Recall-Latency 权衡：}
\begin{itemize}
\item HNSW 在高召回率（>0.97）场景下表现最优，延迟仅需 132734 μs（16线程）
\item IVF-SIMD 通过调整 nprobe 参数可实现 recall 从 0.92 到 0.9996 的灵活切换
\item IVF-PQ 在牺牲少量召回率（~0.97）的情况下，延迟大幅降低
\end{itemize}

\item \textbf{pthread vs OpenMP：}
\begin{itemize}
\item OpenMP 在大多数场景下性能优于 pthread（优势 15\%-34\%）
\item pthread 在 HNSW 场景下表现更优，16 线程时加速比达 6.43×
\item OpenMP 的动态调度对不均匀工作负载的适应性更好
\end{itemize}
\end{enumerate}

\subsection{实验收获}

通过本实验，深入理解了：
\begin{enumerate}
\item ANNS 算法的基本原理和优化方法
\item 多线程并行编程的两种主要范式：pthread 的细粒度控制和 OpenMP 的便捷并行
\item 性能分析与优化的系统方法：使用 perf 工具进行 Profiling
\item 并行程序中的负载均衡和同步问题
\end{enumerate}

\subsection{不足与展望}

\textbf{不足之处：}
\begin{itemize}
\item 未进行更细粒度的参数调优
\item 未测试超过 16 线程的场景
\item 未实现 GPU 并行版本
\end{itemize}

\textbf{未来展望：}
\begin{itemize}
\item 探索混合并行策略：结合 SIMD + 多线程 + GPU 的三层并行
\item 研究学习式索引（Learned Index）以进一步提升 ANNS 性能
\item 在更大规模数据集（如 DEEP1M）上验证算法有效性
\end{itemize}
